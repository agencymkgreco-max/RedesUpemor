Red Campus Segura con VLANs

Orden lógico de configuración, comandos completos y método para leer cualquier diagrama en el examen.

Regla de oro: configura siempre de infraestructura → servicios. Sin VLANs no puedes hacer trunks. Sin trunks VTP no propaga nada. Sin VTP distribuido no puedes enrutar inter-VLAN. Haz clic en cada paso para ver los comandos completos.
01
VTP Configurar VTP ▼
S1 → servidor  |  S2, S3 → clientes
Establece quién administra las VLANs y las distribuye automáticamente.
¿Por qué primero? VTP es el "mensajero" de las VLANs. Si creas VLANs antes de configurarlo, los clientes no las recibirán. El dominio y la contraseña deben coincidir exactamente en todos los switches.
¿Cuándo NO lo configuro? Si el diagrama tiene un solo switch, o todos los switches son independientes sin VLANs compartidas.
S1 — Servidor VTPSwitch(config)#
S1(config)# vtp domain UPEMOR ! mismo nombre en TODOS los switches S1(config)# vtp mode server ! puede crear/borrar VLANs S1(config)# vtp password cisco ! misma contraseña en todos S1(config)# vtp version 2 S1(config)# end S1# wr
S2 y S3 — Clientes VTPSwitch(config)#
S2(config)# vtp domain UPEMOR S2(config)# vtp mode client ! solo recibe VLANs, NO puede crearlas S2(config)# vtp password cisco S2(config)# vtp version 2 S2(config)# end S2# wr
show vtp status show vlan brief
02
VLANs Crear VLANs ▼
Solo en S1 (servidor VTP)
Define los segmentos lógicos. VTP las propagará a S2 y S3 automáticamente.
¿Por qué después de VTP? Al crear VLANs en el servidor, VTP incrementa su "Configuration Revision" y las distribuye a los clientes por los trunks.
¿Cuándo NO creo múltiples VLANs? Si el diagrama tiene una sola red sin segmentación lógica.
S1 — Creación de VLANs (5 VLANs)Switch(config)#
S1(config)# vlan 10 S1(config-vlan)# name ADMIN S1(config-vlan)# vlan 20 S1(config-vlan)# name DOCENTES S1(config-vlan)# vlan 30 S1(config-vlan)# name ALUMNOS S1(config-vlan)# vlan 40 S1(config-vlan)# name LABS S1(config-vlan)# vlan 50 S1(config-vlan)# name INVITADOS S1(config-vlan)# end S1# wr
show vlan brief
03
Trunks Configurar Trunks 802.1Q ▼
Todos los switches — puertos switch↔switch y switch↔router
Las "autopistas" que transportan múltiples VLANs etiquetadas entre dispositivos.
¿Por qué ahora? Sin trunks las VLANs no viajan entre switches ni llega la info VTP a los clientes. Cable switch↔switch o switch↔router = trunk. Cable switch↔PC = access.
¿Cuándo NO es trunk? Puertos a PCs o impresoras son siempre access.
S1, S2, S3 — Trunks entre switchesSwitch(config)#
Sx(config)# interface range gi0/1-2 ! ajustar según modelo del switch Sx(config-if-range)# switchport mode trunk Sx(config-if-range)# switchport trunk allowed vlan 10,20,30,40,50 Sx(config-if-range)# no shutdown Sx(config-if-range)# end Sx# wr
S1 — Puerto trunk hacia R1 (router-on-a-stick)Switch(config)#
S1(config)# interface gi0/0 ! puerto que va al router S1(config-if)# switchport mode trunk S1(config-if)# switchport trunk allowed vlan 10,20,30,40,50 S1(config-if)# no shutdown S1(config-if)# end S1# wr
show interfaces trunk show interfaces status
04
STP Spanning Tree — Root Bridge ▼
S1 → root primary  |  S2 → root secondary
Previene bucles de capa 2. Con S1–S2–S3 en triángulo, STP bloqueará un puerto automáticamente.
¿Por qué después de los trunks? STP actúa sobre los enlaces troncales. Sin trunks no hay topología de bucle que proteger. PVST crea una instancia STP por VLAN.
¿Cuándo NO configuro root manualmente? Si solo hay 2 switches (sin bucle posible), STP funciona solo. Solo configura root cuando hay 3+ switches formando un triángulo.
S1 — Root PrimarySwitch(config)#
S1(config)# spanning-tree mode pvst S1(config)# spanning-tree vlan 10,20,30,40,50 root primary ! prioridad → 24576 S1(config)# end S1# wr
S2 — Root Secondary (respaldo)Switch(config)#
S2(config)# spanning-tree mode pvst S2(config)# spanning-tree vlan 10,20,30,40,50 root secondary ! prioridad → 28672 S2(config)# end S2# wr
show spanning-tree show spanning-tree summary
05
Port-Security Puertos de PC + Port-Security ▼
S2 y S3 — puertos que conectan a PCs (fa0/1, fa0/2…)
Asigna cada puerto a su VLAN, activa PortFast + BPDU Guard y limita cuántas MACs puede aprender.
¿Por qué aquí? Las VLANs ya llegaron por VTP, así que podemos asignarlas. Port-Security requiere que el puerto sea access primero. Sticky aprende MACs con tráfico: haz ping primero para generar ARP.
¿Cuándo NO pongo Port-Security? Solo en puertos access (PCs). NUNCA en trunks. Si la práctica no menciona seguridad de puertos, omítelo.
S2 / S3 — Puerto de acceso completo (repetir por puerto)Switch(config)#
Sx(config)# interface fa0/1 Sx(config-if)# switchport mode access ! PRIMERO: definir como access Sx(config-if)# switchport access vlan 10 ! asignar a su VLAN Sx(config-if)# switchport nonegotiate ! deshabilita DTP ! ── PortFast + BPDU Guard ────────────────────── Sx(config-if)# spanning-tree portfast ! forwarding inmediato (solo en PCs) Sx(config-if)# spanning-tree bpduguard enable ! err-disable si recibe BPDU ! ── Port-Security ────────────────────────────── Sx(config-if)# switchport port-security Sx(config-if)# switchport port-security maximum 2 Sx(config-if)# switchport port-security mac-address sticky Sx(config-if)# switchport port-security violation restrict Sx(config-if)# no shutdown Sx(config-if)# end Sx# wr
show port-security show port-security interface fa0/1 show port-security address
06
Router Router-on-a-Stick + OSPF ▼
R1 → subinterfaces (gateways VLANs)  |  R1, R2, R3 → OSPF
Permite comunicación entre VLANs (R1) y que los tres routers compartan rutas automáticamente.
¿Por qué al final? Para que el router reciba tráfico etiquetado de las VLANs, el trunk entre S1-R1 ya debe existir. Cada subinterfaz = gateway de una VLAN.
¿Cuándo NO hago Router-on-a-Stick? Si hay switch L3, o si solo hay una VLAN.
¿Cuándo NO hago OSPF? Si solo hay un router, o la práctica pide rutas estáticas.
R1 — Subinterfaces una por VLANRouter(config)#
R1(config)# interface g0/0 R1(config-if)# no shutdown ! CRÍTICO: sin esto todas las subif quedan down R1(config)# interface g0/0.10 R1(config-subif)# encapsulation dot1Q 10 R1(config-subif)# ip address 192.168.10.1 255.255.255.0 R1(config)# interface g0/0.20 R1(config-subif)# encapsulation dot1Q 20 R1(config-subif)# ip address 192.168.20.1 255.255.255.0 R1(config)# interface g0/0.30 R1(config-subif)# encapsulation dot1Q 30 R1(config-subif)# ip address 192.168.30.1 255.255.255.0 R1(config)# interface g0/0.40 R1(config-subif)# encapsulation dot1Q 40 R1(config-subif)# ip address 192.168.40.1 255.255.255.0 R1(config)# interface g0/0.50 R1(config-subif)# encapsulation dot1Q 50 R1(config-subif)# ip address 192.168.50.1 255.255.255.0 R1(config-subif)# end R1# wr
R1 — OSPF (VLANs + enlaces a R2/R3)Router(config)#
R1(config)# router ospf 1 R1(config-router)# router-id 1.1.1.1 ! ── enlaces punto a punto hacia R2 y R3 ──────── R1(config-router)# network 10.0.12.0 0.0.0.3 area 0 R1(config-router)# network 10.0.13.0 0.0.0.3 area 0 ! ── redes de VLANs ───────────────────────────── R1(config-router)# network 192.168.10.0 0.0.0.255 area 0 R1(config-router)# network 192.168.20.0 0.0.0.255 area 0 R1(config-router)# network 192.168.30.0 0.0.0.255 area 0 R1(config-router)# network 192.168.40.0 0.0.0.255 area 0 R1(config-router)# network 192.168.50.0 0.0.0.255 area 0 ! ── passive en subinterfaces (no forma vecinos con PCs) R1(config-router)# passive-interface g0/0.10 R1(config-router)# passive-interface g0/0.20 R1(config-router)# passive-interface g0/0.30 R1(config-router)# passive-interface g0/0.40 R1(config-router)# passive-interface g0/0.50 R1(config-router)# end R1# wr
R2 y R3 — OSPF básicoRouter(config)#
R2(config)# router ospf 1 R2(config-router)# router-id 2.2.2.2 R2(config-router)# network 10.0.12.0 0.0.0.3 area 0 R2(config-router)# network 10.0.23.0 0.0.0.3 area 0 ! si R2-R3 están conectados R2(config-router)# end R2# wr ! R3: igual con router-id 3.3.3.3

🔍 Las 7 preguntas para leer cualquier diagrama
01
¿Cuántos switches hay y cómo se conectan?
Si hay 2+ switches entre sí → necesitas trunks. Si forman triángulo → configura STP. El switch con más conexiones = core (VTP Server).
02
¿El cable conecta a switch o a PC?
Switch ↔ Switch o Switch ↔ Router → trunk. Switch ↔ PC → access port. Esto define el tipo de cada puerto.
03
¿Hay grupos o etiquetas de VLAN?
Si hay diferentes grupos de PCs separados → configura VTP + VLANs. Si todas las PCs están en la misma red → no necesitas VLANs múltiples.
04
¿Las PCs de distintas VLANs deben comunicarse?
Si sí → necesitas Router-on-a-Stick (o switch L3). El router conectado al switch core con un trunk tendrá subinterfaces.
05
¿Cuántos routers hay?
Si hay 2+ routers conectados entre sí → configura OSPF. Si solo hay 1 router → no hay vecinos, no necesitas OSPF.
06
¿Piden seguridad de puertos?
Si la práctica dice "limitar MACs" o "proteger de switches no autorizados" → Port-Security + BPDU Guard. Si no lo mencionan, omítelo.
07
¿Qué switch tiene más conexiones?
El más central = S1 core (VTP Server, STP Root Primary). Los que solo tienen PCs = switches de acceso (VTP Client).
⚠️ Reglas de qué NO configurar
NO pongas Port-Security en trunks
Port-Security solo funciona en puertos access. En un trunk da comportamiento inesperado o simplemente falla el comando.
NO pongas PortFast en trunks
PortFast es solo para hosts finales (PCs). En trunks puede causar bucles al saltarse los estados de STP.
NO crees VLANs en switches VTP Client
En modo client el switch solo recibe VLANs. Solo créalas en el servidor (S1). En modo client el comando se rechaza.
NO uses passive-interface en enlaces entre routers
passive-interface impide formar vecinos OSPF. Úsalo solo en subinterfaces hacia VLANs, nunca en los enlaces punto a punto R1↔R2.
